1. CONTEXTE PRODUIT
==================================================

Projet : PredictFinance

Stack actuelle :
- Front : Angular 21
- API : ASP.NET Core .NET 10
- IA : Python

Vision produit cible :
L’utilisateur doit pouvoir :

1. lancer une analyse sur une valeur
2. obtenir une lecture globale du marché
3. voir les patterns candidats possibles
4. voir un classement ou une priorisation de ces patterns
5. sélectionner un pattern
6. voir à quelle phase de ce pattern le marché semble se situer
7. obtenir ensuite un conseil métier dérivé côté API
8. ne jamais confondre :
   - analyse du marché
   - interprétation de pattern
   - conseil produit final

Contraintes importantes :
- l’IA ne doit pas donner directement Buy/Sell/Hold
- le conseil final doit rester dérivé côté API .NET
- le front doit rester passif sur la logique métier critique
- les domaines fermés doivent être transportés via enums/codes stables backend
- le front mappe les codes vers labels/couleurs/badges
- on ne veut pas exposer d’erreurs techniques Python/.NET au front
- on veut une base maintenable plusieurs années

2. ETAT ACTUEL DU PROJET
==================================================

Etat fonctionnel actuel :
- le seul pattern réellement opérationnel aujourd’hui est DOUBLE_TOP
- le moteur Python a été recentré :
  - il ne produit plus directement de recommandation buy/sell/hold
  - il produit surtout un score/probabilité + un contexte de pattern
- l’API .NET dérive maintenant le conseil métier Buy/Sell/Hold
- le front affiche séparément :
  - analyse IA
  - conseil produit
- le système gère déjà les erreurs Python via un contrat JSON inter-process
- les builds/tests passent actuellement

Limites connues :
- le training Python actuel reste essentiellement mono-pattern
- le modèle apprend un signal encore très centré DOUBLE_TOP
- le label actuel n’est pas encore une vraie probabilité métier globale
- la phase du pattern n’est pas prédite par ML, elle est calculée par logique explicable
- l’architecture réelle reste trop marquée mono-pattern
- une partie des contrats et projections risque de mentir si on essaie trop tôt de faire croire à du vrai multi-pattern

3. ORIENTATION D’ARCHITECTURE DEJA TRANCHEE
==================================================

Ces arbitrages sont déjà pris et doivent être respectés dans la proposition cible.

### 3.1 Rôle de l’IA
- l’IA est conservée dans la cible
- l’IA ne doit pas être le pilier central du produit
- l’IA doit être conçue comme un provider interchangeable
- la V1 doit pouvoir exister sans dépendance forte à l’IA
- Python ne doit jamais être la source de vérité métier finale
- Python ne décide jamais Buy/Sell/Hold
- Python peut scorer, prioriser, enrichir, filtrer
- .NET garde la souveraineté métier

### 3.2 V1 possible sans IA
Une V1 forte sans IA est considérée comme saine et acceptable.
Cette V1 peut reposer sur :
- snapshots de marché figés
- détecteurs heuristiques explicables
- phase calculée par règles
- score heuristique
- classement de candidats
- moteur de décision côté API
- historisation complète
- front passif

### 3.3 Direction cible
Le cœur du produit ne doit pas être “Python”.
Le cœur du produit doit être :

- Market snapshots fiables
- Pattern analysis core explicable
- Scoring layer interchangeable
- Decision engine métier souverain
- Persistence / audit / observability solides
- API claire
- Front passif

### 3.4 Mutual exclusivity
Il ne faut pas partir du principe que les patterns sont mutuellement exclusifs.
On ne veut pas forcer par défaut une distribution fermée où la somme ferait 100%.
Plusieurs patterns peuvent être simultanément plausibles.
La direction à privilégier est :
- scoring / ranking
- pas softmax global forcé

### 3.5 NO_PATTERN
NO_PATTERN ne doit pas être traité comme une simple classe plate équivalente aux autres patterns.
La direction à privilégier est :
- mécanisme d’abstention
- statut de type `NO_DOMINANT_PATTERN`
- éventuellement d’autres statuts comme `INSUFFICIENT_DATA`, `MODEL_UNAVAILABLE`, `MARKET_CHANGED`

### 3.6 Sémantique des scores
Il ne faut pas tout appeler “probabilité”.
La proposition cible doit distinguer proprement les natures de score possibles, par exemple :
- pattern match score
- confirmation score
- actionability score
- autre score de ranking
Le système doit nommer clairement le type de score au lieu de mélanger tous les sens dans un seul nombre ambigu.

### 3.7 Snapshot temporel
Le snapshot doit être figé à l’étape 1.
Le détail de pattern doit réutiliser le même snapshot par défaut.
Le refresh doit être explicite.
Il ne doit jamais y avoir de recalcul silencieux entre l’analyse globale et le détail si cela change l’état de marché analysé.

### 3.8 Phase de pattern
La phase doit rester explicable.
La direction à privilégier est :
- phase spécifique par pattern
- éventuellement un stage/stade plus global et transversal en complément
- pas de phase opaque pilotée par ML en V1/V2

### 3.9 Conseil métier
Buy/Sell/Hold ne doit pas être le noyau métier durable.
Il peut exister comme façade simplifiée exposée au front.
Mais le moteur interne doit être plus riche avec une notion de posture métier, par exemple :
- OBSERVE
- WATCH_FOR_CONFIRMATION
- ENTRY_CANDIDATE
- EXIT_CANDIDATE
- NO_ACTION

### 3.10 Source de vérité et historisation
La vraie source de vérité doit être le batch global d’analyse rattaché à un snapshot de marché figé.
Il faut historiser au minimum :
- le snapshot utilisé
- le batch global
- tous les candidats
- les scores
- les détails calculés
- les décisions produites
- les versions de règles / modèles

1. Mutual exclusivity des patterns

- pourquoi ce n’est pas une vraie classe métier plate

5. Phase de pattern
- pourquoi garder la phase explicable
- comment modéliser phase spécifique + stage global si nécessaire
- comment éviter une explosion de complexité avec plusieurs patterns

6. Conseil métier
- pourquoi Buy/Sell/Hold est trop pauvre comme noyau métier
- comment distinguer posture interne et action simplifiée
- comment préparer un moteur de décision plus riche sans surcomplexifier V1

7. Evolutivité
- qu’est-ce qui casse en premier quand on passe de 1 à plusieurs patterns
